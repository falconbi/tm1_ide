TM1 Debug — String Parameter in Arithmetic Bug
================================================

SYMPTOM
-------
  Process runs fine normally (Run button).
  Debug run (temp clone) fails at the MOD expression line:
    "Syntax error on or before: pFYStart + 12, 12) invalid operator"

ROOT CAUSE
----------
  pFYStart was declared as a String parameter (Type: String).
  The temp process clones the parameter list via the TM1 REST API.
  TM1's parser rejects a String-typed declared parameter when used
  directly in arithmetic operations (subtraction, addition) inside
  expressions like MOD().

  The source process (created via TM1 UI/Architect) does NOT have
  this issue. Only REST API-created processes are affected.

FIX
---
  Wrap the String parameter in StringToNumber() before arithmetic:

    nPriorFis = MOD(nCalMonth - StringToNumber(pFYStart) + 12, 12) + 1;

LESSON
------
  When debugging a process clone created via REST API, check parameter
  TYPES first. The REST API may enforce stricter type checking than
  processes created through the TM1 UI. A String parameter used in
  arithmetic will cause a PARSE error (not runtime) in the temp clone
  but work fine in the original process.

  Previous investigation wrongly blamed a TM1 POST parser bug with
  variable name length. That was a red herring — the real issue was
  type coercion.
