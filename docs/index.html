Connecting to a Server

The server list is defined in config/servers.json. Add a server entry with name, host, and port. The IDE proxies all TM1 REST API calls through PAW — no direct TM1 connection is made from the browser.

Select a server from the dropdown in the top bar. Your selection is persisted across sessions in config/forge.json along with your open tabs.

Auth: Login uses your PAW credentials. The session cookie and CSRF token are cached server-side and reused for all TM1 calls.

Rules Editor

Open from the Explorer under any cube. The editor uses Monaco with tm1rules syntax highlighting.

Formatting

Click Format in the toolbar to format the entire rules file. To format a single rule, highlight it first — the formatter applies only to the selection. The formatter preserves regions, comments, and feeders structure.

Validation

Check Rules sends the current buffer to the TM1 tm1.CheckRules endpoint and displays errors inline. Errors are shown in the panel below the editor with line numbers.

Cell Trace

The Trace panel lets you enter element coordinates and see the full calculation chain — which rules fired, in what order, and what intermediate values were produced.

Regions

Use #Region Name / #EndRegion to fold and organise sections. The editor respects region fold markers.

Snippets

Type a function name and press Tab to expand. Common functions (DB, ATTRS, CONTINUE, SKIPCHECK, etc.) have full signature snippets with tab stops.

TI Process Editor

Open from the Explorer under Processes. The editor splits into four tabs: Prolog, Metadata, Data, Epilog.

Parameters

The Parameters panel defines input parameters with name, type, and default value. All IDE-generated parameters follow the p prefix convention (pServer, pCube, etc.).

Running a process

Click Run or press Ctrl+Enter. The process runs via tm1.ExecuteWithReturn which returns timing and the error log filename inline. Results and any errors appear in the panel below.

Debugger

Set breakpoints with F9. The debugger injects capture statements around each breakpoint and runs a modified copy of the process. Variable values at each breakpoint are shown in the debug panel.

Datasource

The Datasource tab configures the TI data source — type (None, ASCII, ODBC, TM1CubeView, TM1DimensionSubset), file path, delimiter, and header row settings.

Dimension Editor

Open from the Explorer under any dimension. Shows the default hierarchy tree on the left and the attribute grid on the right.

  • Add / rename / delete elements — right-click context menu in the tree
  • Add edge — drag an element onto a consolidation, or use the context menu
  • Attributes — edit inline in the grid; add/delete attribute columns via the column header menu
  • Bulk import — paste a tab-separated list of elements with optional attributes

View Editor

Open from the Explorer under any cube → Views. Supports both native TM1 views and MDX views.

Native views

Drag dimensions between the Row, Column, and Title (filter) axes. Click a dimension pill to assign a subset — choose a named subset or define member selections inline. The cell grid updates live as you change the view structure.

MDX views

Switch to MDX mode to write a raw MDX query. The result grid renders the same as native views. MDX views can be saved back to the server.

Writeback

Click any numeric cell in the grid to edit it. Changes are written via tm1.Update. The cell flashes green on success.

Subset Editor

Open from the Explorer under any dimension → Subsets, or from within the View Editor when assigning a subset to an axis.

  • MDX mode — write an MDX set expression directly; preview executes it and shows the member list
  • Visual mode — browse the hierarchy tree and tick members to build a static subset
  • Guided filter — build an MDX filter by attribute value, level, or parent without writing MDX

Save as Static (fixed member list) or MDX (expression evaluated at runtime).

Chore Editor

Open from the Explorer under Chores. Shows the schedule, step list, and execution controls.

  • Schedule — set start time, frequency (daily, weekly, monthly, etc.), and DST handling
  • Steps — add/remove/reorder TI processes; set parameters per step
  • Activate / Deactivate — toggle the chore without changing its schedule
  • Execute Now — run the chore immediately outside its schedule

SQL Editor

Connects to external databases (MSSQL, PostgreSQL, MySQL, SQLite) — useful for querying source systems before writing TI to pull from them.

  • Connection definitions stored in config/sql-connections.json
  • Schema browser shows tables and columns on the left
  • Saved queries stored in config/sql-queries.json
  • Post to TI — copies the query as a TI datasource configuration snippet

Change Sets

A Change Set is a named session that tracks every save you make — rules, TI processes, dimensions, views, subsets. Without an active Change Set, changes are still saved to TM1 but are not included in a deploy package.

Start a Change Set

Click Start Change Set in the status bar. Give it a name (e.g. "Sprint 14 — Budget fixes"). All subsequent saves are logged against this session.

Close a Change Set

Click Close in the Change Set panel. The session is marked complete and can be selected as the source for a deploy.

Note: Only saves made while a Change Set is active are tracked. Saving without an active Change Set shows a toast reminder.

Deploy Pipeline

Open the Deploy tab from the toolbar. Select a closed Change Set as the source and a target server to deploy to.

1

Diff — compares each changed object against the target server. Shows what has changed, what is new, and what is missing on target.

2

Package — fetches the live objects from the source server and writes them to disk as a deployable package with a manifest.json.

3

Risk — checks for blockers: syntax errors in rules, missing cube dependencies on target, chore conflicts. Blockers must be resolved before deploy.

4

Deploy — pushes each object to the target server in dependency order. Run as a Dry Run first to validate without making changes, then click Deploy for Real.

Supported object types

  • Rules (cube calculation rules)
  • TI Processes
  • Dimensions (elements, edges, attributes)
  • Cubes (structure — create only, not destructive)
  • Views (native and MDX)
  • Subsets (static and MDX)
  • Element attributes
Deploy order: Objects are always deployed in dependency order — attributes → dimensions → cubes → rules → subsets → views → processes.
